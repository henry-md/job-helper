- Symptom: after confirming overwrite for an already completed tailored resume, the side panel could briefly return to the old completed row and sibling panels could jitter between completed and loading states before the backend published the new run.
- Root cause: overwrite only set local preparation/prompt state while leaving the old prompt and saved-resume fallback in shared storage/cache until the capture flow reconciled. Other panel instances could rehydrate that stale completed state during the backend delay.
- Fix: confirming overwrite now immediately clears the shared overwrite prompt, publishes an optimistic running record with the old tailored-resume/application suppression ids, and writes the filtered personal-info snapshot to the shared cache before the backend request resolves.
- Guardrail: replacement flows must publish the same scoped running state and fallback-suppression identifiers that sibling panels render; do not wait for server refresh to hide the replaced saved artifact.
