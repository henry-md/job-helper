- Symptom: the Tailor Resume UI could show duplicate in-flight cards or open a new job tab even when the matching Workday posting was already open with tracking query parameters.
- Root cause: comparable job URL handling did not normalize Workday's canonical URL shape against the browser URL shape, and active/saved Tailor Resume lists trusted every local/server row instead of collapsing rows that pointed at the same comparable posting URL.
- Fix: normalize Workday job URLs to the locale/career-site/requisition path before hashing or comparing, prefer the actual browser URL when capturing a page for tailoring, dedupe visible applications/tailored resumes/active runs by comparable URL, and let Tailor Resume reuse older application rows whose stored hash came from the previous URL shape.
- Current guardrail: job-posting comparisons are now query-sensitive exact URL comparisons because some employers, including Microsoft, put job-description identity in query params. Do not reintroduce query stripping, Workday alias collapsing, or nested-path containment for Tailor Resume association without an explicit product decision.
