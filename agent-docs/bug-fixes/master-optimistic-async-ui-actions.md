# Master: optimistic async UI actions

- Problem: long-running actions such as Tailor Resume start, Stop run, delete, archive, restore, and navigation can block unrelated buttons when the UI ties global disabled state to one pending request. This makes the extension feel frozen and creates races where stale local state repaints after the user already stopped or replaced a run.
- Principle: user-visible state should update optimistically at the narrowest safe scope, while backend work continues asynchronously. Disable only the exact control or artifact being mutated unless a modal intentionally owns the whole surface.
- Tailor start guardrail: publish a visible local preparation or running record before auth, page capture, overwrite checks, or backend generation can delay UI feedback. When overwriting, immediately hide the old completed fallback and show the new loading card.
- Stop/cancel guardrail: abort local requests, clear local run/interview/preparation state, and re-enable the panel immediately. Send backend cancellation in the background and reconcile later. A delayed cancel request must not keep `Tailor Current Page`, row menus, archive buttons, or unrelated run controls disabled.
- Per-card action guardrail: background run actions must track state by card id, not with one shared `stopping`/`deleting` flag. Stopping one run should not disable another run's menu or saved-resume row actions.
- Saved artifact guardrail: archive/restore/delete should optimistically remove or update only the targeted row. Other rows and unrelated tabs should remain actionable unless a confirmation dialog is actively open.
- Failure guardrail: if a background request fails after an optimistic update, restore the previous snapshot for the affected scope only, show a scoped error, and avoid dropping the whole panel into a loading state.
- Storage guardrail: cleanup should target captured job registry keys and normalized job identity, not ambient active-tab focus. Late storage events from aborted or stale requests must not resurrect cleared UI.
- Regression checks to run when touching this area: current run Stop with delayed backend cancel, background run Stop with delayed backend cancel, delete one saved resume with a slow backend response, archive/restore one row while another remains clickable, overwrite confirmation to loading card, tab switching while a run is active, and stale local storage reconciliation after server state disagrees.
