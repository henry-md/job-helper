- Symptom: the extension could show a Tailor Resume card with completed Step 4 copy such as `Tailored resume ... is ready.` while the preview/review actions were still unavailable in the same card.
- Root cause: the success copy and completed progress state trusted the persisted `lastTailoringRun`, but the preview/review affordances waited for a second source of truth: the refreshed `personalInfo.tailoredResumes` list. During that refresh gap, the run already had a saved `tailoredResumeId`, but the UI still behaved as if nothing reviewable existed.
- Fix: derive one shared "reviewable tailored resume id" for the current card. Prefer synced completed-tailoring matches first, but fall back to the saved `lastTailoringRun.tailoredResumeId` while the list is catching up, and suppress that fallback once the synced summary explicitly shows the resume archived.
- Guardrail: completion copy, compact-complete card state, preview/review affordances, and deep links should all read from the same resolved tailored-resume id so the UI never claims "ready" before it has a concrete item it can open.
