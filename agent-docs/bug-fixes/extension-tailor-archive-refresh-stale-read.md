- Symptom: archiving a tailored resume from the extension could succeed on the server, but after the side panel reloaded the resume still appeared unarchived or the `Archived` tab looked empty.
- Root cause: the extension rehydrates `personalInfo` through the background script immediately after archive/restore mutations. Those background `fetch()` calls were allowed to reuse the browser HTTP cache, so a cached pre-mutation `/api/tailor-resume` or `/api/job-applications` response could overwrite the fresh archive state that the PATCH had just returned.
- Fix: force the extension background refresh reads for `/api/tailor-resume`, `/api/job-applications`, and `/api/sync-state` to use `cache: "no-store"` so archive/restore reconciliation always replays server truth instead of a cached snapshot.
