- Symptom: after a Tailor Resume run completed from the side panel, reopening the in-page Job Keywords popup could show the initial scraped bucket guesses instead of the reviewed saved keyword classifications.
- Root cause: the background script kept an in-memory active-run keyword snapshot from the first Step 1 scrape event. Side-panel completion updated local run storage to `success`, but did not clear that background-only memory, so later tab badge refreshes could prefer stale active keywords over the completed tailored-resume badge.
- Fix: when a remembered active keyword snapshot matches a stored run that is no longer active, drop the remembered snapshot and let the completed resume badge render. Completed saves also preserve the reviewed Step 1 keyword snapshot when storing the final planning result, so planner-filtered keyword lists cannot become the display source.
- Guardrail: completed-resume keyword badges must never be sourced from active-run memory once the matching local run is `success` or `error`. Starting or overwriting a Tailor Resume run must also clear all matching local run/preparation/prompt entries for the page identity, not just the exact registry key, so URL aliases cannot replay previous-run data.
