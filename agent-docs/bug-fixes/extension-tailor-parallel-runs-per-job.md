- Symptom: starting Tailor Resume on one job could disable the tailor action or hijack the visible run state for a different tab, which effectively forced the extension into a single global queue instead of letting jobs progress in parallel.
- Root cause: the extension stored transient Tailor Resume progress in one global set of `chrome.storage.local` keys and the backend exposed only one pending interview, so tab switches and concurrent jobs all competed for the same client and server slot.
- Fix: scope extension run/preparation/overwrite-prompt state by normalized job/page identity, track abort controllers per page key, and store pending interviews as an array so each job can keep its own active generation or Step 2 interview without blocking unrelated tabs.
- Related UX fix: when a run pauses on Step 1, keep the summary card visible and show an explicit `Open chat` action instead of auto-opening the interview UI, so multi-job runs do not steal focus when the user is browsing other tabs.
- Guardrail: Tailor Resume concurrency should be keyed by job identity, not by "current active tab" or one global run slot; only same-job restarts should conflict with each other.
