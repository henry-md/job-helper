# Master: cross-tab and web-app sync

- Problem: one extension panel, another extension panel, and the web app can each mutate the same user-visible application and Tailor Resume state. If the clients depend only on local React state, active-tab focus, or eventual manual refresh, one surface can show a loading run, deleted row, restored row, or web-deleted artifact that no longer matches the rest of the product.
- Principle: every user-visible mutation must publish a shared invalidation or snapshot immediately, then reconcile with server truth. Sibling tabs should update from shared extension storage, and web-app changes should update through the lightweight sync-version poll.
- Latency guardrail: visible extension panels should observe shared local extension changes immediately and server-originated sync-version changes within one poll interval. The target budget is no later than 1 second after the originating mutation or detected web-app version bump.
- Shared local-state guardrail: optimistic extension mutations must write the same scoped state to the shared personal-info cache or Tailor run registry that the originating panel renders. Do not update only the current component tree when the result is visible in another tab.
- Web-app guardrail: server mutations that affect extension-visible data must bump the correct sync-state counter. Extension panels should poll the small sync-state endpoint while visible and fetch personal-info only when the version changes.
- Current-page guardrail: only current-page affordances, such as row highlighting and active-tab page capture, may depend on the active tab. Global lists, active runs, archived rows, deletes, and restore/archive state must come from shared storage or server state.
- Failure guardrail: if a fresh server fetch fails after a sync-version change, invalidate the changed stale slices rather than repainting known-stale rows. If an optimistic extension mutation later fails, restore or refresh only the affected slice and publish that correction through the shared cache.
- Regression checks to run when touching this area: two extension panels see a newly loading tailoring run within 1 second, a stop/delete in one panel clears the matching run or row in another panel within 1 second, archive/restore/delete row mutations broadcast to sibling panels without disabling unrelated controls, and a simulated web-app sync-version bump removes or refreshes stale extension rows within 1 second.
