- Symptom: a Tailor Resume run or saved tailored resume could appear in the side panel on the tab where it started, then disappear when the user switched to another tab, even though the unarchived Tailor surface is supposed to be global apart from the current-page highlight.
- Root cause: the extension refreshed server-backed Tailor state from the current tab's `lastTailoringRun`, so a different side-panel instance could stay stale if that tab had no local run record. The panel also relied on one-time storage hydration plus live `chrome.storage.onChanged` delivery, which was not enough to guarantee every tab-scoped panel instance caught up after tab switches. A second bug then made stale completed resumes linger even after the shared state refreshed: the panel could synthesize a finished tailored resume from local `lastTailoringRun` state, and it ignored refresh payloads whose `tailoredResumes` array was present but empty.
- Fix: drive the heavier Tailor Resume refresh from the shared per-job run registry instead of the current tab's local run, rehydrate the shared Tailor registries again whenever the active tab changes or the panel becomes visible, and treat the background-owned personal-info cache as a shared cross-tab snapshot. Tab switches should hydrate from that shared cache immediately, while the background refreshes it again when a run actually starts, reaches interview follow-up, or finishes. Finished tailored resumes must render only from synced personal-info data, never from local run fallbacks, and explicit empty tailored-resume payloads must clear the cached list instead of preserving old rows.
- Guardrail: global Tailor Resume UI must sync from shared storage/server state across all tabs; only current-page affordances such as the blue highlight should depend on the active tab.
