- Symptom: overwriting a completed tailored resume from the extension could leave the old saved resume visible during the replacement run, then save the replacement as a second completed row instead of replacing the old one.
- Root cause: overwrite only sent a generic overwrite flag to the API, so the backend could miss the specific saved profile/DB artifact when job URL variants or application links did not line up exactly. The extension visibility filter also suppressed fallbacks only by resume id or comparable URL, not by application id.
- Fix: carry `applicationId` through active/completed tailoring state, send the old tailored-resume/application target with overwrite requests, and remove old profile/DB/PDF artifacts only after the replacement generation succeeds. While a replacement is still active, suppress saved fallback rows by application id, explicit old resume id, or comparable URL.
- Guardrail: overwrite flows should keep old artifacts as backend fallback until the replacement is usable, but every active/completed UI surface must treat the active replacement as owning that job/application so stale saved rows do not leak through.
