- Symptom: after overwriting a completed Tailor Resume run, the new Step 1 scraped keyword badge could appear on the job page and then be replaced by the old completed resume's keyword list during a later tab badge refresh.
- Root cause: badge refresh only trusted the shared personal-info payload. If that payload was cached or briefly behind the local streaming run, it could resolve the previous completed tailored resume and redraw its keywords even though a replacement run was active.
- Fix: hide stale in-page badges when a new run is dispatched, and let badge refresh consult the local active run registry before completed resumes. A matching active local run now either replays its streamed Step 1 keywords or suppresses the completed badge until fresh keywords arrive.
- Guardrail: in-page completed-resume badges must never win over a matching active local tailoring run; overwrite/stop handoffs should prefer the newest active run state while server/cache reconciliation catches up.
