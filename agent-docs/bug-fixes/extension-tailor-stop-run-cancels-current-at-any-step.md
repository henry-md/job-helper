- Symptom: the extension could show a live Tailor Resume run, but there was no single `Stop run` action that reliably canceled the current try across step 1/3/4 generation and the step 2 follow-up chat, and late aborted requests could repaint a failure after the user tried to stop.
- Root cause: cancellation logic was split between `cancelExistingTailoring` and interview-only discard, while in-flight panel and background requests kept streaming or resolving after the UI had already moved on.
- Fix: add a shared `cancelCurrentTailoring` backend action, route the extension `Stop run` control through it, abort any in-flight panel or background request, clear extension-local run/interview state immediately, and ignore late step events or abort errors from the canceled request.
- Guardrail: when a user stops a long-running extension flow, cancellation has to cover both the shared backend run state and any local in-flight request state, or stale step events and failure messages will leak back into the UI after the stop.
