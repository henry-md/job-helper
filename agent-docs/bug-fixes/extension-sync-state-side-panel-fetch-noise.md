- Symptom: the Chrome side panel could log `Failed to fetch` / `ERR_CONNECTION_REFUSED` from its sync-state polling path while the extension was otherwise already using the background worker for authenticated API reads.
- Root cause: the side panel fetched `/api/sync-state` directly instead of routing the request through the extension background worker, so transient local-server connectivity issues surfaced as panel-level network errors and the "preserve current" refresh path could still clear the current Personal snapshot on failure.
- Fix: move sync-state reads onto the background worker's bearer-authenticated request path and let preserve-current refreshes keep the last ready Personal payload when a background reload attempt fails.
- Guardrail: extension UI polling should reuse the background worker for authenticated server reads so auth/session recovery, network failures, and retry behavior stay centralized instead of each panel tab inventing its own fetch path.
