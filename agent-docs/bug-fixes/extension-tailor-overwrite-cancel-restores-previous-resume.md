- Symptom: after starting an overwrite for a job that already had a saved tailored resume, canceling the new run could leave the extension looking idle instead of returning to the previously saved tailored resume for that same page.
- Root cause: the side panel only treated `activeTailoring` or the last optimistic run as sources of truth for a completed tailored resume. Once overwrite cleared those transient objects, the UI stopped deriving the matching saved resume from the persisted tailored-resume list for the current page.
- Fix: when the current page matches a saved completed tailored resume and no overwrite/stop transition is still in progress, let the extension fall back to that page-scoped saved record for the top `Web` action and the completed-review state.
- Guardrail: overwrite and cancel flows may clear transient run state early, so any page-scoped completed UI should be able to rehydrate from persisted saved records instead of assuming the latest local run object still exists.
