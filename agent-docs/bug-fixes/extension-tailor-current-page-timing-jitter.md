- Symptom: the extension Tailor tab could visibly jitter on the same job URL as an in-flight Tailor Resume run. The Step 1 timer could snap back to `0:00` or keep changing after Step 1 had already completed, while the problem was much less visible from unrelated tabs.
- Root cause: the current-page run path mixed three refresh sources: the local extension stream, server-polled active tailoring state, and active-tab recapture. Server refreshes often report only the latest step with `durationMs: 0`; when those updates overwrote local step timings, completed Step 1 lost its measured duration. Tailor run storage changes also recaptured the active tab, briefly remounting current-page state on every run update.
- Fix: route single-step stream merges through the full timing-history merge, preserve the original start time for repeated running steps, compute a terminal step duration when a running step completes with no backend duration, and keep completed step timings stable across later server refreshes. The current-page card now prefers the local transient run over the server echo, Tailor run storage changes refresh run registries without recapturing the page, and the background keeps timing history when the run moves into Step 2 chat.
- Guardrail: current-page Tailor UI should treat server active-tailoring data as a supplement to the local stream, not a replacement for local timing history. Storage-driven run updates should not call active-tab capture unless the page itself changed.
