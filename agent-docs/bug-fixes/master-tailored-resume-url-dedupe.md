# Master: tailored resume URL dedupe

- Problem: tailored resumes can surface from multiple stores at once: profile JSON, DB rows, active run rows, pending interviews, extension local storage, extension personal-info cache, and job applications. If each surface compares raw URLs independently, the UI can show duplicate rows for one job, highlight both an active run and an old saved fallback, or replace the wrong record after generation finishes.
- Principle: dedupe by a stable job identity, not by display text or raw URL. Prefer `applicationId` when present, then a normalized exact job URL, then explicit tailored-resume ids when suppressing a known fallback artifact.
- Canonical normalization: use the shared normalizers in `lib/tailor-resume-job-url.ts` on the app/server side and `extension/src/comparable-job-url.ts` in the extension. Keep their semantics in sync. Normalization removes hashes, forces http/https equivalence, lowercases hostnames, and trims trailing slashes, but it must preserve the full query string and must not collapse Workday aliases or nested apply paths. Microsoft can include job-description identity in query params, so current-page highlighting, badge display, storage lookup, and backend URL hashes are query-sensitive.
- Canonical suppression model: when an active replacement exists, pass all useful identifiers to visibility code: `applicationId`, active run/interview id, the overwritten tailored-resume id, current page URL, canonical URL, job URL, stored run URL, preparation URL, and saved fallback URL. The backend may retain the old fallback until the new run succeeds, but user-visible lists should hide that fallback while the replacement is active.
- Backend guardrail: final overwrite cleanup must happen only after a new usable tailored resume exists. Until then, keep the old artifact as backend fallback and hide it in UI through suppression references.
- Frontend guardrail: current-page active run cards must win over completed fallback cards. Do not let a hidden completed fallback continue to power preview/review/top-level actions while a replacement run is visible.
- Ordering guardrail: always sort duplicate candidates newest-first before filtering, then keep one record per application key or comparable URL. Do not rely on API insertion order.
- Regression checks to run when touching this area: query-distinct Microsoft URLs, saved duplicate URL rows, duplicate rows with shared `applicationId`, active replacement hiding an old saved fallback, active local run plus server active run dedupe, current-page highlighting only the exact URL, and post-success cleanup removing old backend artifacts.
