Tailor Resume skills-only tool keywords

- Symptom: concrete job tools such as Jenkins or Confluence could be extracted and classified as skills-section terms, then still fail to appear in the tailored Skills section.
- Root cause: Step 3 filtered emphasized technologies using resume/USER.md text presence before planning. That deterministic gate was too conservative for Skills placement because adjacent concrete skills should remain available to the model even without bullet-level proof. The saved support query tools also returned only spare-bullet evidence, which made durable skills-only support invisible in tool output.
- Fix: remove pre-planning experience/support filtering so Step 3 sees the full non-skill keyword set, return `skillsOnlySupport` from single/batch skill query tools, and prompt Step 3/4 to treat named developer, CI/CD, and collaboration tools as normal Skills/Technical Skills candidates.
- Guardrail: do not add deterministic filtering that removes job-emphasized concrete skills before planning based on whether they appear in experience text. Use prompt guidance instead: Skills candidates are the superset of job-emphasized terms adjacent to existing resume skills plus anything in USER.md or first-class skill support. Skills-only support is direct permission for Skills/Technical Skills placement; it is not a requirement to invent a body bullet.
