Chrome extension auth:
- The extension does not connect to Postgres directly. It obtains a Google access token with Chrome's identity API and sends it to `POST /api/extension/auth/google`.
- The server verifies the token audience against `GOOGLE_EXTENSION_CLIENT_ID`, reads Google userinfo, links or creates the normal `google` `Account`, and creates a standard NextAuth database `Session`.
- Extension API calls send the returned session token as `Authorization: Bearer <token>`. Server routes that need extension access should use `getApiSession(request)` instead of calling `getServerSession(authOptions)` directly. When a bearer token is present, it is authoritative over ambient browser cookies.
- `GET /api/tailor-resume` returns the authenticated user's Tailor Resume profile, including all `profile.tailoredResumes`; extension callers should read this endpoint with the bearer session token instead of trying to read profile storage or Postgres directly.
- When the extension needs its personal-info snapshot, prefer one enriched `GET /api/tailor-resume` response that includes the small application summary payload the side panel renders instead of making a second client-side `/api/job-applications` read just to leave loading state.
- `GET /api/sync-state` returns the authenticated user's latest cross-surface sync versions; the extension may poll this lightweight endpoint every second while visible and should only call the heavier application/profile endpoints when one of those versions changes.
- Tailor Resume calls from the extension should send the captured posting URL as `jobUrl` and page-derived `applicationContext` alongside the job-description text. The API uses that URL to create/reuse a tracked `JobApplication`, store live `TailorResumeRun` state, and return a conflict prompt when that application already has an active run or linked tailored resume.
- `GET /api/job-applications` returns the authenticated user's tracked applications for app and extension clients; extension callers should use the same bearer token and keep display-specific summaries in extension code.
- The extension personal-info tab fetches the original rendered resume preview from `GET /api/tailor-resume/preview` with the bearer token, then renders the returned PDF as an extension-local object URL.
- Extension chat calls should stream directly from an authenticated app API with the bearer session token. Keep the extension responsible for collecting the active page context and rendering streamed chunks; keep URL-scoped history, prompt/context assembly, resume plaintext, `USER.md`, and model calls on the server.
- Extension workflows should stay thin: the extension captures page context and calls authenticated app APIs; the server should own app data access, profile storage, and prompt/pipeline logic.
- When Tailor Resume pauses for follow-up questions, the extension should read `profile.workspace.tailoringInterview`, render that conversation in the side panel, and continue with `PATCH /api/tailor-resume` action `advanceTailorResumeInterview`. Background-triggered runs must treat `tailoringStatus: "needs_user_input"` as a pending sidebar conversation, not as a finished resume.
- When Tailor Resume returns `tailoringStatus: "existing_tailoring_found"`, the extension should keep the user in the side panel and ask whether to cancel/keep or overwrite the existing run/resume instead of showing a generic error.
- Opening protected browser pages from the extension goes through `POST /api/extension/auth/browser-session`, which returns a short-lived encrypted handoff URL. The handoff route sets the normal NextAuth session cookie and redirects to a same-origin callback path.
- No Prisma migration is needed for the basic bridge because it reuses the existing `User`, `Account`, and `Session` tables.
- Google Cloud needs two OAuth clients: the existing web client for NextAuth callback redirects and a separate Chrome Extension client whose Item ID matches the loaded extension id from `chrome://extensions`.
- `GOOGLE_EXTENSION_CLIENT_ID` must be available to the server, while `VITE_JOB_HELPER_APP_BASE_URL` must be available at extension build/dev time. The extension Vite config loads env from both the repo root and the extension directory.
- A missing visible OAuth popup is not necessarily a failure. Chrome can silently reuse an existing Google profile/grant, so the side panel should show the connected Google avatar/email to make the active account visible.
- If Chrome reports that the token audience is wrong, first compare the loaded extension id with the Google OAuth client's Item ID. If the unpacked id is drifting, set a stable `CHROME_EXTENSION_PUBLIC_KEY` before rebuilding.
